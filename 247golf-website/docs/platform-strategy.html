01 Executive Summary

Our website must scale as fast as our business

Growth is outpacing our platform. We open 5 new locations before October 1 and target 50+ by 2030 — but every new location page is hand-built in WordPress + Elementor.

Editing is the bottleneck. Location owners can't safely update their own pages, and the AI tooling in the WordPress/Elementor ecosystem falls far short of what we can do with Claude Code.

Our top competitor proves the model. The Back Nine runs a fully custom, component-based site — fast, premium, and consistent — the experience we want at scale.

Recommendation: move to a Git-based, structured-data website (Astro + GitHub + Cloudflare Pages) run Claude Code-first: a new location is one data file, the location finder updates itself, and owners edit with their own Claude Code — or a friendly CMS.

03 Where We Want To Go

What the next platform must deliver

Onboard a location in hours, not weeks

A new location page — on-brand and complete — should be generated from a simple set of facts: name, address, bays, owner bio, photos, CourtReserve link.

Owner self-service, safely

Every location owner edits their own page — hours, events, photos, memberships — without touching anyone else's page or the global design. HQ reviews before anything goes live.

A location finder that updates itself

Adding a location to the finder map and list should happen automatically when the location is added — never a separate manual step.

AI-native editing with Claude Code

Our best editing experiences today come from Claude Code. The platform should treat AI-assisted, code-and-content workflows as the primary path, not a workaround.

04 Competitor Benchmark

The Back Nine: the experience bar we're chasing

thebackninegolf.com — our primary competitor and the site we admire most in the category

Premium, cohesive design — one tightly art-directed system. Typography, motion, and imagery feel like a national brand, not a template.

Fast and lightweight — purpose-built pages with only the code each page needs, served through Cloudflare's global CDN.

Booking woven into the site — booking is a first-class part of the experience (their own "b9-golf-book" application), not a link out to a third-party portal.

The takeaway

Back Nine's advantage isn't a better page builder — it's that they left page builders behind. Their site is generated from structured components and data. That's the architecture we should adopt.

07 Option 1

Stay on WordPress — but re-architect it

If we keep WordPress, we must stop hand-building pages and make locations structured data.

How it works

  • Create a "Location" custom post type: every location page is generated from structured fields (address, bays, owner, hours, photos, CourtReserve link) instead of being built by hand
  • The location finder reads from the same data — adding a location automatically adds it to the map and list
  • Replace or reduce Elementor with a lightweight block theme for speed and consistency; optionally move to Elementor Hosting to unlock their AI tools
  • Give each owner a restricted editor role scoped to only their location
  • Keep our existing Zapier + CourtReserve automations untouched
Strengths
  • No platform migration — lowest disruption
  • Familiar admin for owners and staff
  • Large plugin ecosystem
Trade-offs
  • AI-assisted editing remains far behind Claude Code
  • Plugin and security maintenance burden continues
  • Page-builder lock-in and performance overhead persist
  • We'd be investing further in the platform we've outgrown
VerdictViable and cheapest short-term — but it modernizes the process without fixing the platform. A fallback, not a destination.

08 Option 2

Astro + GitHub + Cloudflare PagesRecommended

A modern static-first site where every location is one data file — and Claude Code is a first-class editor.

How it works

  • Site lives in a new private GitHub repo under the lud247golf account, seeded from the 247GolfWebsiteV2 scaffold (design system and Claude Code context files already built)
  • Each location = one structured data file. The build generates the location page and its finder map/list entry automatically — onboarding a location takes minutes
  • Owners edit with their own Claude Code subscription connected to the repo — or through a free Git-based CMS (simple forms) for those who prefer no code
  • Every change gets a private preview link; HQ approves before it merges and goes live. Instant one-click rollback
  • Cloudflare Pages publishes globally on the same CDN Back Nine uses — hosting cost is effectively zero
Strengths
  • Best-in-class Claude Code / AI workflow
  • Fastest possible pages; scales to 500 locations as easily as 5
  • Built-in review, previews, and version history
  • Near-zero hosting cost; no plugins to patch
Trade-offs
  • Upfront build effort (mitigated: repo and design system already started)
  • Owners must learn a new — simpler — editing interface
  • Custom features require development rather than plugins
VerdictThe same architecture that makes Back Nine's site excellent — achievable for us because Claude Code does the heavy lifting. The next section shows how we'd run it day-to-day.

09 Option 2 · The Operating Model

Claude Code-first, with HQ guardrails

One repo, one AI workflow: HQ owns the frame, each owner owns their page, and Claude Skills keep everyone on-brand.

How it runs

  • Initial build by Eric with Claude Code: the full site plus all 10 location pages, in the new private lud247golf repo
  • Each owner connects their own Claude Code subscription to the repo. Branch protection means owners propose changes; HQ reviews the preview and merges — nobody can break the live site
  • Clear boundaries: the global header menu and footer are HQ-controlled; each location page — memberships, events, photos, and an optional secondary menu — belongs to its owner
  • Shared Claude Skills in the repo (.claude/skills) teach every owner's Claude Code the site's components, rules, and boundaries automatically
  • CourtReserve membership skill — working today: one command pulls a location's live plans, prices, and signup links from CourtReserve and rebuilds the on-brand "Choose Your Membership" section, never inventing a price. On the new platform it publishes as a reviewed page update
Why it's strong
  • One AI-native workflow for HQ and owners — no second toolset to build or maintain
  • Skills encode the guardrails, so owner edits stay on-brand by default
  • Owners' own Claude subscriptions are the only per-location software cost
  • Every change is reviewed, previewed, and instantly reversible
Risks to manage
  • Owners must buy and learn Claude Code — adoption will vary by owner
  • Repo permissions and branch protection must be airtight from day one
  • Eric is the early bottleneck for reviews and owner support
  • Mitigation: Git-based CMS forms remain the no-code fallback for any owner
VerdictMake Claude Code the primary owner editing path, with the CMS as the safety net — the model that best fits our team, our tooling, and our growth rate.

10 Option 3

Next.js + headless CMS (Sanity or Payload)

A full web application platform — the choice if the website itself becomes a product.

How it works

  • React-based application framework with a hosted visual editing studio for content
  • Locations are structured content in the CMS; pages and the finder render from live data
  • Strongest foundation for future interactive features: member portals, live bay availability, deep CourtReserve API integration rendered natively on our site
  • Deploys to Vercel or Cloudflare with preview environments, like Option 2
Strengths
  • Most powerful and future-proof option
  • Excellent live-preview editing experience
  • Great Claude Code compatibility
Trade-offs
  • Highest cost: CMS fees plus ongoing developer involvement
  • More moving parts than a content site needs today
  • Longer initial build
VerdictThe right platform if and when booking and member experiences move onto our site — likely a Phase 2, not the place to start.

11 Option 4

Webflow — a managed visual builder

The polished, hosted successor to the Elementor model.

How it works

  • Modern hosted platform with a professional visual designer and clean output
  • Locations become items in a Webflow CMS Collection; pages and finder entries generate from a template
  • Location owners get "Editor mode" — they can change text and images on their page without design access
  • Webflow handles hosting, CDN, and security entirely
Strengths
  • Beautiful results without code
  • Structured location data with templated pages
  • Zero infrastructure to maintain
Trade-offs
  • Closed platform: no Git, no Claude Code workflow — the exact gap we're trying to close
  • Per-seat and hosting costs grow with 50+ owners
  • Proprietary lock-in; hard to leave later
VerdictA real upgrade over Elementor for a design-led team — but it walks away from our biggest advantage: AI-native editing with Claude Code.

12 Recommendation

Build the platform Back Nine has — the AI-native way

Adopt Option 2 with the Claude Code-first operating model. It is the only option that wins on every criterion that matters: it matches Back Nine's proven architecture, makes onboarding a location a one-file task, turns owner edits into reviewed and reversible changes, and builds on strengths we already have — deep Claude Code expertise, our CourtReserve API stack, and a design system that's already scaffolded. Options 1 and 4 keep us tied to weaker tooling; Option 3 is the right Phase 2 once booking experiences move onto our own site.
